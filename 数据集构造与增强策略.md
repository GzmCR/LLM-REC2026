# 数据集构造与增强策略

本文记录当前对 `dataset/` 来源、原始数据利用情况、以及后续数据增强方向的分析。需要特别区分两类结论：

- **可确认事实**：来自仓库脚本、README、已生成的数据分析报告、以及对本地 JSONL/parquet 的直接检查。
- **基于样本反推的推断**：当前仓库没有提供从 `OneReason_UserProfile` 原始 63 列行为表直接生成 `dataset/懂*.jsonl` 的完整构造脚本，因此这部分只能由样本格式和字段对应关系反推。

## 1. 可确认事实

当前仓库中与数据转换直接相关的脚本有两个：

- ` Explorer_LLM_Rec_Competition/demo/convert_jsonl.py`
  - 输入：已经构造好的 `{system, prompt, response}` JSONL。
  - 输出：LLaMA-Factory/Alpaca 风格 `{instruction, input, output, history}` JSONL。
  - 作用：只是格式转换，不负责从原始用户行为表构造样本。
- ` Explorer_LLM_Rec_Competition/demo/convertv2.py`
  - 输入：包含 `messages` 字段的 parquet。
  - 输出：Alpaca JSONL。
  - 作用：处理 message 对话格式、清理 SID 结束符、补 `<think>` 模式，也不是从 `OneReason_UserProfile` 的 63 列行为特征构造推荐样本。

已分析的 `dataset/` 统计：

- 总样本数：`32,480`。
- `懂推荐*.jsonl`：`19,204` 条，多域历史行为到推荐目标。
- `懂用户.jsonl`：`2,892` 条，用户时间线到主题相关行为抽取或逻辑链。
- `懂物料part*.jsonl`：`10,384` 条，物料描述与 SID token 的双向映射。
- `懂用户.jsonl` 的 `system` 字段为空，其他文件的 `system` 基本非空。

本地原始 parquet 表：

- `OneReason_UserProfile`：用户多域行为序列，本地样本 `50,000` 行，完整字段 `63` 列。
- `OneReason_Pid2Sid`：`(domain, pid) -> sid_three`，用于把哈希 ID 转成三段式 SID。
- `OneReason_Pid2Caption`：`(domain, pid) -> caption`，本地样本只包含 `goods` caption。
- `OneReason_Pid2Tag`：`(domain, pid) -> tag_lv3`，本地样本包含 `video/video`、`video/ad`、`live`。
- `OneReason_General`：通用 reasoning/message 数据，本地样本 `1,000` 行。

## 2. dataset 反推加工链

当前没有官方完整构造脚本，但从样本可以反推出大致流程：

1. 从 `OneReason_UserProfile` 中读取用户多域行为序列。
2. 按 domain 选择主序列：
   - 视频：`video_sampled_pid_list`、`video_history_sampled_pid_list`
   - 商品：`ec_item_id_list`、`ec_colossus_rs_item_id_list`、`ec_good_click_item_id_list_extend`、`ec_good_order_item_id_list_extend`
   - 直播：`live_hist_author_id_list`
   - 广告：`outer_loop_history_action_pid_list_pos`、`outer_loop_history_action_pid_list_click`、`outer_loop_deep_target_pid`
3. 用 `(domain, pid)` join `OneReason_Pid2Sid`。
4. 把 `sid_three = [s_a, s_b, s_c]` 转成文本 token：
   - `video/video` -> `<|video_begin|><s_a_x><s_b_y><s_c_z>`
   - `video/ad` -> `<|ad_begin|><s_a_x><s_b_y><s_c_z>`
   - `goods` -> `<|prod_begin|><s_a_x><s_b_y><s_c_z>`
   - `live` -> `<|living_begin|><s_a_x><s_b_y><s_c_z>`
5. 根据行为字段生成自然语言行为标签：
   - 视频：深度观看、看过、浏览、长播、点赞、评论、转发、收藏等。
   - 商品：浏览、点击、购买等。
   - 直播：关注、首次打赏等。
   - 广告：点击、深度转化等。
6. 用模板或 LLM 改写生成 `{system, prompt, response}`。
7. 加入 `/think`、`/no_think`、`<think>...</think>` 等推理格式。

### 当前三类任务

`懂推荐*.jsonl`：

- 输入：用户多域历史行为，通常包含视频、商品、直播、广告 SID。
- 输出：推荐目标 SID。
- 目标不只视频，也包含商品、广告、直播；但当前“最近喜欢的视频”占比最高。
- 作用：训练模型从多域行为预测下一步目标内容。

`懂用户.jsonl`：

- 输入：按日期组织的用户行为时间线，外加一个主题。
- 输出：JSON 数组形式的相关 SID，或 JSON 对象形式的行为逻辑链。
- 作用：训练模型做用户兴趣挖掘、主题相关行为抽取、跨域行为链解释。

`懂物料part*.jsonl`：

- `part1-4`：描述 -> SID，覆盖商品、直播、广告、视频。
- `part5-7`：SID -> 描述，覆盖商品、广告、视频。
- 作用：训练模型理解 SID 与自然语言物料语义之间的映射。

## 3. 未充分使用的原始字段

下面的“未充分使用”指在当前 `dataset/` 文本中没有显式保留，或只被压缩成了粗粒度行为标签。它不代表官方构造时完全没有用过。

### 3.1 `OneReason_Pid2Tag`

字段：

- `pid`
- `domain`
- `tag_lv3`

价值：

- `tag_lv3` 是非常直接的类目信号，可以把 SID 的离散 token 和可解释语义连接起来。
- 适合用于用户兴趣类目归纳、同类 SID 泛化、负样本过滤、跨域兴趣迁移。

风险：

- 本地 `Pid2Tag` 没有 `goods`，只有 `video/video`、`video/ad`、`live`。
- 不能直接假设 `tag_lv3` 与 `[s_a, s_b, s_c]` 一一对应；它更像辅助语义标签。

### 3.2 视频强反馈字段

字段：

- 当前视频序列：
  - `video_sampled_pid_list`
  - `video_neg_feedback_list`
  - `video_like_list`
  - `video_comment_list`
  - `video_forward_list`
  - `video_collect_list`
  - `video_watch_time_list`
  - `video_play_done_list`
  - `video_duration_list`
  - `video_ts_list`
- 历史视频序列：
  - `video_history_sampled_pid_list`
  - `video_history_neg_feedback_list`
  - `video_history_like_list`
  - `video_history_comment_list`
  - `video_history_forward_list`
  - `video_history_collect_list`
  - `video_history_watch_time_list`
  - `video_history_play_done_list`
  - `video_history_duration_list`
  - `video_history_ts_list`
  - `video_history_seq_len`

价值：

- `watch_time / duration` 可以得到观看比例。
- `play_done` 可标记完播。
- `like/comment/forward/collect` 是强正反馈。
- `neg_feedback` 是明确负反馈。
- `ts` 可做近期兴趣、长期兴趣和兴趣漂移。

风险：

- 长序列非常长，本地 `video_history` p95 约 `1931`，最大超过 `25,000`，不能直接全塞进 prompt。
- 需要做截断、聚合、采样或摘要，否则训练成本和上下文长度都会爆。

- `ec_colossus_rs_is_cart_list`
- `ec_colossus_rs_is_buy_list`

- `ec_trunc_clk_lag`
- `ec_good_order_item_id_list_extend`

价值：

- 本地 `ec_item_id_list` 覆盖很低，但 `ec_colossus_rs_item_id_list`、点击扩展、购买扩展覆盖较高。

- `video_history_ts_list`
- `outer_loop_history_action_pid_list_pos_ts`
- `outer_loop_history_action_pid_list_click_ts`
- `outer_loop_deep_target_pid_ts`

- `ec_colossus_rs_lagv2_list`

价值：

- 可构造近期兴趣、长期兴趣、兴趣漂移、节奏变化。

- 时间信息过细会增加 prompt 噪声，建议分桶：近 1 天、近 7 天、近 30 天、更早。

## 4. 两条数据构造路线

### 路线 A：补充原 `dataset`

保持现有 `{system, prompt, response}` 格式，对 `懂推荐`、`懂用户`、`懂物料` 做增强版本。

适合场景：

- 希望训练流程最少变化。
- 希望直接接入现有 `convert_jsonl.py` 和 LLaMA-Factory 配置。
- 希望做少量消融：原始数据 vs 原始 + 增强。

可以补充的内容：

- 在 `懂推荐` prompt 中追加部分 `tag_lv3`。
- 把视频行为细化为“完播/长播/点赞/收藏/负反馈”。
- 把商品行为细化为“曝光未点/点击/加购/购买”。
- 把广告行为细化为“点击行业/深度转化目标”。
- 把时间线压缩成近期、中期、长期三段。

优点：

- 与当前训练格式兼容。
- 对 baseline 改动小。
- 更容易快速实验。

缺点：

- 容易改变原 dataset 分布。
- 如果增强 prompt 太长，可能反而影响模型学习目标 SID。
- 需要控制混入比例。

### 路线 B：新增辅助 dataset

新增独立 JSONL 文件，仍保持 `{system, prompt, response}`，但任务目标与当前三类任务区分开。

适合场景：

- 希望模型先学会类目、强弱反馈、时序、漏斗、负反馈等基础能力。
- 希望避免直接污染原始 `dataset`。
- 希望做多任务 SFT。

建议新增任务：

1. `增强类目理解.jsonl`
   - `SID -> tag_lv3`
   - `tag_lv3 -> 相关 SID`
   - `用户历史 SID -> 兴趣类目归纳`
2. `增强反馈强度.jsonl`
   - 输入视频/直播/商品/广告行为。
   - 输出强兴趣 SID、弱兴趣 SID、负反馈 SID。
3. `增强时间兴趣.jsonl`
   - 输入按时间分桶的行为。
   - 输出近期兴趣、长期兴趣、兴趣漂移。
4. `增强电商漏斗.jsonl`
   - 输入曝光、点击、加购、购买序列。
   - 输出购买意图、候选排序或相关行为链。
5. `增强广告意图.jsonl`
   - 输入广告点击、行业、深转化。
   - 输出广告行业偏好、深转化目标。
6. `增强直播画像.jsonl`
   - 输入直播互动统计和主播 SID。
   - 输出直播品类偏好、高互动主播、负反馈主播。

优点：

- 任务边界清楚。
- 便于控制每类能力的数据量。
- 后续可以按任务分别做 ablation。

缺点：

- 需要写新的构造脚本。
- 多任务比例需要调。
- 辅助任务不一定直接提升 leaderboard，需要验证。

## 5. 具体构造建议

### 5.1 `tag_lv3` 类目增强

新 dataset 任务：

```json
{
  "system": "你是推荐语义理解助手，需要理解内容SID与三级类目的关系。",
  "prompt": "给定内容token：<|video_begin|><s_a_...><s_b_...><s_c_...>，请判断其三级类目。",
  "response": "游戏娱乐-游戏解说-传奇游戏解说"
}
```

也可以反向构造：

```json
{
  "system": "你是推荐召回助手，需要根据三级类目生成相关内容token。",
  "prompt": "三级类目：游戏娱乐-游戏解说-传奇游戏解说。请给出相关内容token。",
  "response": "<|video_begin|><s_a_...><s_b_...><s_c_...>"
}
```

建议：

- 优先作为新增辅助 dataset。
- 少量样本可混入 `懂推荐` prompt，但不要给每个 SID 都追加类目，避免 prompt 过长。

### 5.2 视频强反馈增强

反馈强度建议：

- 强正：完播、长播、高观看比例、点赞、评论、转发、收藏。
- 弱正：普通观看。
- 负反馈：`neg_feedback = 1`。

新 dataset 任务：

```json
{
  "system": "你是用户视频兴趣分析助手，需要根据行为强度区分强兴趣、弱兴趣和负反馈。",
  "prompt": "用户视频行为：完播 <SID1>，点赞 <SID2>，负反馈 <SID3>，普通观看 <SID4>。请输出强兴趣、弱兴趣、负反馈内容。",
  "response": "{\"strong_interest\": [\"<SID1>\", \"<SID2>\"], \"weak_interest\": [\"<SID4>\"], \"negative\": [\"<SID3>\"]}"
}
```

建议：

- 优先用作辅助 dataset。
- 对 `懂推荐` 可增强行为词，而不是增加大量数值。

### 5.3 电商漏斗增强

行为强度建议：

- 曝光未点击：弱负或未知。
- 点击：弱正。
- 加购：中强正。
- 购买：强正。
- `cvr_label`：转化监督信号。

新 dataset 任务：

```json
{
  "system": "你是电商推荐意图分析助手，需要理解曝光、点击、加购、购买之间的漏斗关系。",
  "prompt": "用户近期商品行为：曝光未点 <SID1>，点击 <SID2>，加购 <SID3>，购买 <SID4>。请判断购买意图最强的商品token。",
  "response": "<SID4>"
}
```

建议：

- 可补充 `懂用户` 的“行为链”任务。
- 商品数据强依赖 SID/caption join，构造时要过滤 join 不上的样本。

### 5.4 时间与 recency 增强

时间分桶建议：

- 近期：近 1 天或最近若干条。
- 中期：近 7 天。
- 长期：更早稳定重复出现的行为。

新 dataset 任务：

```json
{
  "system": "你是用户兴趣时序分析助手，需要区分近期兴趣、长期兴趣和兴趣漂移。",
  "prompt": "长期行为：<SID1>, <SID2>。近期行为：<SID3>, <SID4>。请输出近期兴趣和长期兴趣。",
  "response": "{\"recent_interest\": [\"<SID3>\", \"<SID4>\"], \"long_term_interest\": [\"<SID1>\", \"<SID2>\"]}"
}
```

建议：

- 对长序列先聚合再入 prompt。
- 不建议直接暴露大量精确 timestamp，优先使用相对时间描述。

### 5.5 直播互动增强

直播兴趣强度建议：

- 强正：有效播放、长播放、点赞、评论、关注。
- 负反馈：`reduce_similar`、`report_live`。
- 类别偏好：`author_category_type`、`author_type`。

新 dataset 任务：

```json
{
  "system": "你是直播推荐画像助手，需要根据直播互动识别用户偏好。",
  "prompt": "用户直播行为：长时间观看并关注 <SID1>，点赞评论 <SID2>，减少相似 <SID3>。请输出直播偏好画像。",
  "response": "{\"positive_live\": [\"<SID1>\", \"<SID2>\"], \"negative_live\": [\"<SID3>\"]}"
}
```

建议：

- 直播字段很多，应聚合成少数强信号。
- 可作为 `懂推荐` 中直播目标预测的补充。

### 5.6 广告行业增强

广告意图建议：

- 点击：普通广告兴趣。
- 深度转化：强广告兴趣。
- 行业：兴趣解释。
- 点击类型：意图强弱或行为类型。

新 dataset 任务：

```json
{
  "system": "你是广告推荐意图分析助手，需要根据点击行业和深度转化判断用户广告偏好。",
  "prompt": "用户广告行为：点击行业=教育培训 <SID1>，深度转化 <SID2>。请归纳广告偏好并输出目标广告token。",
```

- 混入比例不要太高，避免模型过度生成行业解释而不是 SID。

### 5.7 `OneReason_General`

建议用途：

- 少量混入，用于保持通用 reasoning、格式稳定、`<think>` 输出能力。
- 不作为推荐主力数据。

建议混入方式：

- 在多任务训练中占比较低，例如先从 `1% - 5%` 尝试。
- 如果 leaderboard 任务非常强调 SID 推荐，优先不混或极低比例混入。

## 6. 推荐优先级

第一优先：新增辅助 dataset，不直接污染原 dataset 分布。

- 先做 `tag_lv3` 类目理解、视频强反馈、电商漏斗、时间兴趣四类。
- 每类先生成小规模样本，跑一版快速 SFT。
- 与原始 `dataset` 做消融对比。

第二优先：少量混入增强版 `懂推荐`。

- 只增强部分样本。
- 控制 prompt 长度。
- 重点补“强反馈”和“近期兴趣”，不要一次性塞入所有字段。

第三优先：验证有效后扩大混合比例。

- 如果新增辅助任务提升稳定，再逐步增加比例。
- 如果发现模型输出格式变差，应降低辅助任务比例，或把辅助任务放在预热阶段。

## 7. 建议实验顺序

1. 保留官方 `dataset` 不动，新增 `generated_dataset/` 存放增强数据。
2. 先实现一个构造脚本，输出小样本：
   - `generated_dataset/tag_lv3_aux.jsonl`
   - `generated_dataset/video_feedback_aux.jsonl`
   - `generated_dataset/ec_funnel_aux.jsonl`
   - `generated_dataset/time_interest_aux.jsonl`
3. 每个辅助集先生成 `1k - 5k` 条。
4. 训练配比先保守：
   - 原始 `dataset`：`80% - 90%`
   - 辅助 dataset：`10% - 20%`
5. 观察：
   - SID 格式是否稳定。
   - 是否更少生成无效 token。
   - 是否更懂强反馈和负反馈。
   - 线上或本地 proxy 指标是否提升。

## 8. 结论

当前 `dataset` 已经覆盖了 SID 生成、用户行为理解和物料理解，但对原始数据中的 **类目、反馈强度、时序、漏斗、负反馈、行业意图** 利用还不充分。

后续最稳妥的方向不是直接改写原始 `dataset`，而是先新增辅助 dataset，让模型补齐这些能力；等确认有效后，再少量混入增强版 `懂推荐` 做消融。这样既能利用原始数据的潜在价值，又能降低破坏原始训练分布的风险。
